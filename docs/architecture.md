# Архитектура

`datalens-dev-mcp` — локальный stdio MCP-сервер с исходящим клиентом DataLens API. MCP-клиент управляет жизненным циклом процесса. Сервер не открывает HTTP-порт.

## Поток данных

```text
Codex / Claude / другой MCP-клиент
  -> stdio JSON-RPC
  -> 39 инструментов MCP
  -> чтение / планирование / валидация / Safe Apply
  -> DataLens Public API

project root
  <-> требования, снимки, планы, контрольные чтения, отчёты

DATALENS_ENV_FILE
  -> ID организации, IAM-токен, write/save/publish-настройки
```

Stdout зарезервирован для MCP JSON-RPC. Логи и диагностика пишутся в stderr. Значения учётных данных не включаются в ответы и созданные файлы.

## Слои пакета

| Путь | Ответственность |
| --- | --- |
| `src/datalens_dev_mcp/server.py` | Инициализация MCP и диспетчер инструментов, prompts и resources |
| `src/datalens_dev_mcp/mcp/` | Контракты и реализации публичных инструментов |
| `src/datalens_dev_mcp/api/` | Авторизация, схемы методов, транспорт и очищенные ошибки |
| `src/datalens_dev_mcp/pipeline/` | Режим задачи, планы, Safe Apply, контрольные чтения и отчёты |
| `src/datalens_dev_mcp/editor/` | Сборка и проверка объектов Editor |
| `src/datalens_dev_mcp/validators/` | Проверка маршрутов, payload, SQL, связей, секретов и URI |
| `src/datalens_dev_mcp/knowledge/` | Компактный поиск по справочным данным DataLens |
| `src/datalens_dev_mcp/assets/` | Схемы, правила и шаблоны, входящие в Python-пакет |

## Границы безопасности

- Локальная папка проекта и live ID DataLens задаются отдельно.
- План не считается выполненной записью.
- Saved и published версии читаются и проверяются отдельно.
- Статическая проверка, чтение API и проверка интерфейса являются разными видами подтверждения.
- Перед записью сверяются цель и ревизия; публикация строится из saved readback.
- Произвольное удаление целого объекта недоступно; manifest action
  `retire_legacy_objects` требует отдельного подтверждения.

Подробности: [модель безопасности](local-only-safety-model.md), [Safe Apply](safe-apply.md), [выбор технологии](route-policy.md) и [контракты ответов](mcp/response_contracts.md).

## Производительность

- Все API-клиенты одного процесса и endpoint используют общий request-start limiter. Встроенный интервал `1.05` секунды соответствует примерно 57 стартам в минуту; до трёх независимых чтений могут выполняться одновременно. HTTP 429 устанавливает общий `Retry-After` cooldown, а запись и fresh-read/write/save/readback/publish остаются эксклюзивными.
- Безопасные чтения локально повторяются не более двух раз после timeout, connection reset/close и HTTP 502/503/504. 404 терминален, а неоднозначная запись автоматически не повторяется. IAM refresh выполняется single-flight с коротким negative cooldown после сбоя.
- Manifest пакетных runtime-ресурсов вычисляется один раз на процесс. Override через `DATALENS_MCP_RESOURCE_ROOT` всегда сканируется заново.
- Повторный `dl_snapshot_dashboard` сначала заново читает выбранные ветки дашборда и inventory воркбука. Полная гидратация связей, чартов, датасетов и подключений пропускается только при неизменном полном векторе ревизий и совпадающих SHA-256 сохранённых артефактов. При полной гидратации независимые charts, datasets и connections читаются параллельно по стадиям с сохранением детерминированного порядка. Отсутствующая ревизия, изменение цели или повреждённый файл приводят к полной гидратации.
- Dashboard full/debug readback строится из проверенного snapshot manifest и не перечитывает те же charts второй раз. Batch workbook read сохраняет порядок и изолирует item-level ошибки.
- Packaged JSON/JSONL/text, source trace и Editor-валидация кэшируются по содержимому и версии правил. Файловый resource override остаётся некэшируемым.
- Повторный `dl_validate_project` в том же процессе использует отчёт только при точном совпадении metadata-fingerprint дерева файлов. Любое добавление, изменение или удаление файла сбрасывает результат.
- Неизменившиеся локальные артефакты не перезаписываются. Тяжёлые планы и workflow возвращают summary примерно до 15 КБ и ссылку с SHA-256 на полный результат; вложенные delivery-summary содержат только статус, счётчики, блокировки и пути к readback.
- Очистка одного вложенного ответа вычисляет набор секретов окружения один раз и применяет его ко всем строкам объекта; правила и шаблоны редактирования не меняются.

Эти ускорения не кэшируют fresh read перед записью, revision guard, saved readback, publish-from-saved или published readback.

## Поставка

Поддерживаемый формат поставки — Python-пакет с локальным stdio entrypoint. Все необходимые runtime-ресурсы входят в пакет. MCP-клиенты используют одинаковые command, args и env независимо от своего интерфейса.
