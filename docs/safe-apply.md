# Защищённое применение изменений

**Русский** · [English](safe-apply_en.md) · [Сценарии](usage-flow.md) · [Безопасность](local-only-safety-model.md)

Safe Apply связывает исходную задачу пользователя, точную цель, актуальную ревизию, запрос к DataLens API и результаты контрольного чтения.

## Последовательность

1. Прочитать целевой объект и его связи.
2. Проверить тип объекта, ID и сохранённую ревизию.
3. Составить изменение поверх актуальной версии.
4. Проверить payload и проектные материалы.
5. Создать plan с нормализованным режимом, target lock и SHA-256 исходной задачи.
6. Непосредственно перед записью снова проверить write/save/publish-настройки.
7. Выполнить save.
8. Прочитать saved-версию и сверить результат.
9. Если задача требует публикации, сначала собрать и проверить publish-действия
   для всей группы из saved readbacks.
10. Выполнить всю проверенную publish-группу и прочитать published-версии.
11. Для видимых изменений проверить изменённый объект в интерфейсе DataLens.

Явная команда создать, исправить, обновить, улучшить или переработать известный объект авторизует обычные save и publish в рамках этой последовательности. Повторный вопрос перед ними не задаётся.

## Режимы задачи

| Формулировка | Результат |
| --- | --- |
| audit, review, diagnose, inspect | Чтение и отчёт |
| plan-only, dry-run | План без записи |
| save-only, no-publish, draft | Save и saved readback |
| create, fix, update, enhance, redesign | Save, saved readback, publish-from-saved, published readback |
| manifest action `retire_legacy_objects` | План удаления и отдельное подтверждение |

Режим сохраняется в плане и наследуется при построении publish-from-saved. Publish не запускается, если исходная задача была `save-only` или `no-publish`.

## Plan и target lock

`dl_create_safe_apply_plan` создаёт файл плана внутри project root. План содержит:

- нормализованный режим операции;
- SHA-256 исходного пользовательского запроса;
- тип и точный ID цели;
- ожидаемую ревизию и branch;
- изменяемые поля и желаемый overlay;
- методы API и ожидаемые контрольные чтения;
- результаты проверок и причины остановки.

`dl_execute_safe_apply` получает `plan_path`. Перед RPC он перечитывает цель, проверяет совпадение target lock и накладывает overlay на актуальную версию. Идентификаторы, ревизия, технология чарта и неизвестные поля сохраняются.

## Жёсткие выключатели

- `DATALENS_MCP_ENABLE_WRITES=0` блокирует все запросы записи.
- `DATALENS_MCP_LIVE_ALLOW_SAVE=0` блокирует save.
- `DATALENS_MCP_LIVE_ALLOW_PUBLISH=0` разрешает доступный save и завершает цикл как `saved_not_published`.

Эти значения проверяются непосредственно перед соответствующим RPC. Они имеют приоритет над формулировкой задачи и local config.

## Контрольное чтение

Режимы `readback_mode`:

- `minimal` — целевой объект и данные, необходимые для проверки изменения;
- `full` — все указанные связанные объекты;
- `debug` — полный набор данных для диагностики;
- `none` — только для операции без live-записи с явным объяснением.

Saved и published читаются отдельно. Отчёты и файлы этих этапов имеют разные имена и не перезаписывают друг друга.

## Публикация из сохранённой версии

Обычный update собирает publish из saved readbacks внутри
`dl_execute_safe_apply`. До первого publish RPC сервер проверяет saved artifact
каждого объекта и весь action set. Ошибка подготовки одного объекта блокирует
публикацию всей группы.

`dl_create_publish_from_saved_plan` — явный resume-инструмент. Он принимает
saved readback, тип и ID объекта, а также необязательный
`target_workbook_id`. План фиксирует `revId`, `savedId` и путь к сохранённому
результату. Публикация блокируется, если saved readback отсутствует, устарел
или относится к другой цели.

После публикации `dl_readback_and_report` читает published-версию и создаёт deployment report. Для изменения интерфейса API-readback подтверждает структуру, а проверка DataLens подтверждает отображение.

## Обновление существующего объекта

При update желаемые изменения хранятся отдельно от актуального объекта. Непосредственно перед записью сервер накладывает их на свежую saved-версию. Для дашборда изменение содержимого сохраняет координаты нетронутых виджетов; изменение layout должно явно описывать ожидаемую геометрию.

Группа связанных действий сначала полностью сохраняется и проверяется. Публикация группы начинается после успешного saved readback всех затронутых объектов.

### Семантическое объединение диапазона дат

`dl_create_safe_apply_plan` принимает
`maintenance_contract.kind=date_range_selector_merge` для точечного обновления
существующего Editor-селектора и его mount на дашборде. Компилятор заменяет
ровно два статических `datepicker` одним `range-datepicker`, меняет только две
строковые array-value в Params, синхронизирует dashboard defaults и сохраняет
остальные controls, layout, technology и неизвестные поля.

План содержит отсортированный `target_objects` lock для selector+dashboard,
`workflow_metrics` с лимитом 14 RPC и обязательный `runtime_smoke`. До browser
smoke результат изменения помечается `runtime_smoke_required`.

## Конфликты и неопределённый результат

- Блокировка объекта и нарушение уникальности возвращаются как конфликт без повторной записи.
- Устаревшая ревизия требует нового чтения и нового плана.
- Ошибка после отправки записи без подтверждённого результата возвращает `write_outcome_unknown`; перед продолжением выполняется чтение текущего состояния.
- Повтор создания после прерывания начинается с `dl_reconcile_partial_creates`, чтобы не создавать дубликаты.

## Удаление целого объекта

Произвольное удаление целого объекта не входит в стандартный lifecycle surface.
Единственный поддерживаемый путь — объявленное в project manifest действие
`retire_legacy_objects`. Первый вызов возвращает
`delete_confirmation_required`, точные ID, связи и hash плана. Выполнение
требует повторного вызова с `confirm_delete=true` и тем же plan. Любое
изменение цели или плана аннулирует подтверждение. Удаление целого QL-объекта
не поддерживается.

Удаление легенды, фильтра, колонки, вкладки или виджета внутри объекта является update и проходит обычные проверки записи.
